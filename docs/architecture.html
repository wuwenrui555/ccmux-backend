ccmux — backend architecture

A standalone Python library that mirrors Claude Code sessions running inside tmux into a small, stable Python API. Zero knowledge of any chat platform.

What it is

ccmux watches tmux panes and Claude Code's JSONL transcripts, tracks tool_use / tool_result pairing, and exposes a single Backend Protocol that any frontend (Telegram bot, CLI, web UI) can drive. It owns three state files under ~/.ccmux/ and provides a ccmux hook CLI that Claude Code's SessionStart hook calls on every new session.

External dependencies are minimal:

libtmux aiofiles python-dotenv

Public API — ccmux.api

Every symbol a frontend is allowed to see is re-exported from this one module. Everything else is internal and may change without notice.

The Protocol

Split into two sub-Protocols by domain. Tmux operations on backend.tmux, Claude-JSONL operations on backend.claude. Cross-domain queries and lifecycle stay at the top level.

class TmuxOps(Protocol):
    async def send_text(window_id, text) -> tuple[bool, str]
    async def send_keys(window_id, keys: list[str]) -> None
    async def capture_pane(window_id) -> str
    async def create_window(cwd, session_name=None) -> str
    async def list_windows() -> list[TmuxWindow]

class ClaudeOps(Protocol):
    async def list_sessions(cwd) -> list[ClaudeSession]
    async def get_history(session_id, *, start_byte=0,
                          end_byte=None) -> tuple[list[dict], int]

class Backend(Protocol):
    tmux: TmuxOps
    claude: ClaudeOps

    # Cross-domain queries
    def is_alive(window_id) -> bool
    def get_window_binding(window_id) -> WindowBinding | None

    # Lifecycle
    async def start(on_message, on_status) -> None
    async def stop() -> None

Call style: await backend.tmux.send_text(wid, "hi"), await backend.claude.get_history(sid), backend.is_alive(wid).

Event payloads (callbacks)

ClaudeMessage

Passed to on_message. Fields: session_id, role, content_type, text, tool_use_id, tool_name, image_data, timestamp, is_complete.

WindowStatus

Passed to on_status. Fields: window_id, window_exists, pane_captured, status_text, interactive_ui. No frontend routing fields.

InteractiveUIContent

Nested in WindowStatus. Fields: content, name (the matched pattern label).

Query / listing types

WindowBinding

Returned by backend.get_window_binding(). Fields: window_id, session_name, claude_session_id, cwd.

ClaudeSession

Returned by backend.claude.list_sessions(). Fields: session_id, summary, message_count, file_path.

TmuxWindow

Returned by backend.tmux.list_windows(). Fields: window_id, cwd, pane_current_command.

Parsing helpers public

Used by the frontend when it capture_panes and needs to parse the result locally (status line extraction, interactive-prompt detection, /usage modal, !-command output). Re-exported so frontends don't duplicate the regex zoo.

  • TranscriptParser — JSONL → ClaudeMessage parser, also handles tool pairing
  • extract_bash_output(pane_text, command) — extract !-command output from a pane
  • extract_interactive_content(pane_text) — detect AskUserQuestion, ExitPlanMode, Permission, BashApproval, etc. (returns None if no prompt; use is not None for boolean check)
  • parse_status_line(pane_text) — spinner + working text above chrome
  • parse_usage_output(pane_text) — /usage modal parser

Composition

  • DefaultBackend(tmux_registry, window_bindings, ...) — the built-in implementation; frontend constructs it in its own main.py
  • tmux_registry — process-wide TmuxSessionRegistry singleton
  • TmuxSessionRegistry / WindowRegistry — exposed for composition
  • get_default_backend() / set_default_backend(b) — for call sites that can't thread a backend handle through (e.g. Telegram handlers)
  • sanitize_session_name(name, existing) — make a string a legal tmux session name

State files

Under ~/.ccmux/ by default (override with CCMUX_DIR):

FileWritten byShape
window_bindings.json ccmux hook CLI {tmux_session_name: {window_id, session_id, cwd}}
claude_monitor.json MessageMonitor {tracked_sessions: {session_id: {file_path, last_byte_offset}}}

The frontend's own telegram_tmux_map.json is not owned by ccmux — it lives in the ccmux_telegram package, keyed by (user_id, thread_id).

Data flow

Claude → consumer (messages)

Claude writes JSONL ↓ MessageMonitor.poll() reads new bytes incrementally ↓ TranscriptParser.parse_entries() stitches tool_use ↔ tool_result ↓ DefaultBackend fast loop fires on_message(ClaudeMessage) ↓ Frontend consumer routes by session_id / window_id

Terminal pane → consumer (status)

Fast poll tick (every ~0.5s) ↓ StatusMonitor iterates WindowRegistry.all() ↓ For each window_id: tm.capture_pane() + terminal parsers ↓ Emit WindowStatus (no user_id / thread_id fields) ↓ on_status(WindowStatus) fires; frontend resolves routing

User command → Claude (outbound)

Frontend receives user input ↓ backend.tmux.send_text(window_id, text) ↓ WindowRegistry.send_to_window delegates to TmuxSession ↓ TmuxSession.send_keys writes to the pane; Claude reads it

Session lifecycle (hook)

Claude Code starts up ↓ SessionStart hook fires → `ccmux hook` subprocess ↓ Hook writes {session:window_id → session_id, cwd} to window_bindings.json ↓ Backend slow loop reloads the map; verify_all() picks up the new window

What ccmux is not

  • Not a bot. No Telegram, Slack, HTTP, or WebSocket code. No message rendering, no markdown, no rate limiting.
  • Not stateful across topics / users / threads. All identity is window_id / session_id. If a frontend wants "whose turn is this", that's the frontend's problem.
  • Not a transport layer. It doesn't know how on_message / on_status get delivered — they're plain Python callbacks.
  • Not configurable by caller. No pluggable parsers, no strategy patterns. Swap the whole Backend implementation if you need different semantics.

Versioning promise

  • Anything re-exported from ccmux.api is stable. Signature or semantic changes require a major bump.
  • Anything reachable only via ccmux.<submodule> is internal. Changes ship in minor or patch releases.
  • State-file schemas (window_bindings.json, claude_monitor.json) may evolve in minor releases with a best-effort migration; breaking changes trigger a major.
